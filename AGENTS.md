# 🤖 Правила для ИИ-агентов: smith-windows

## 🎯 Контекст
`smith-windows` — API для вызова инструментов автоматизации Windows через `uiautomation` crate.
Цель: надёжное, тестируемое, модульное основание для UI-автоматизации.

## 🛠️ Стек
- Rust 1.95, `tokio`, `thiserror`/`anyhow`, `serde`
- UIAutomation 0.24.4: `uiautomation`
- Тесты: встроенные `#[test]`, `cargo clippy -- -D warnings`

## 📐 Правила
1. **Сначала спека → потом `/plan` → потом код**. Не меняй контракты без согласования.
2. **Модульность**: один файл = одна ответственность.
3. **Без `unwrap()`/`panic!`** в библиотеке. Только `Result`/`Option`.
4. **Идемпотентность**: повторный вызов с теми же данными не ломает состояние.
5. **UIAutomation-first**: все UI-операции через `uiautomation` crate.

## 📁 Источники (порядок чтения)
1. `docs/design/<module>/specification.md`
2. `docs/design/<module>/contract.md`
3. `docs/design/<module>/test-plan.md`
4. `docs/design/<module>/brief.md`
5. `docs/adr/<N>-<module>.md`
6. `AGENTS.md`, `ARCHITECTURE.md`
7. `context_bundle.md` — эталонный контекст проекта (генерируется через `tools/bundle_context.rs`)

**Примечание**: Перед началом разработки нового модуля запустите сбор контекста:
```bash
cargo run --bin bundle_context
```

## 🗓️ Работа с `/plan` (ОБЯЗАТЕЛЬНО)
1. После получения спеки/контракта → построй план через `/plan`
2. План должен включать: файлы, типы, функции, тесты, проверки
3. Не переходи к генерации кода без утверждения плана архитектором
4. Если план неполный → задай уточняющие вопросы ДО исполнения

## 🧪 Тесты
- На каждую публичную функцию: минимум 1 позитивный + 1 негативный тест
- Граничные случаи (0, пусто, максимум) — обязательны
- `cargo test` и `cargo clippy` должны проходить без предупреждений

## ⚠️ Запреты
- ❌ `unwrap()`, `expect()`, `panic!` в `src/`
- ❌ Глобальные мутабельные состояния без `Arc` + явного контекста
- ❌ Изменение контракта без обсуждения
- ❌ Генерация кода без утверждённого плана `/plan`
- ❌ `CoInitializeEx`, `CoCreateInstance`, `BSTR`, `VARIANT` в бэкендах
- ❌ `GetForegroundWindow()` — использовать `is_enabled()`/`is_offscreen()` через `uiautomation`

## 🔄 Процесс
1. Архитектор пишет спеку + контракт + план тестов → `docs/design/<module>/`
2. Планировщик создаёт `/plan` → архитектор утверждает
3. Кодер генерирует код + тесты строго по плану
4. Проверка: `cargo test && cargo clippy -- -D warnings`
5. Решение логируется в `docs/adr/`
